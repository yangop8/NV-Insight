## 1. OpenAI GPT-5.5 运行于 NVIDIA GB200 NVL72 基础设施

【快讯内容】
OpenAI 发布 GPT-5.5 并驱动新一代 Codex，全部在 NVIDIA GB200/GB300 NVL72 系统上训练和推理。双方完成首个 10 万 GPU GB200 NVL72 集群联调，每百万 token 成本降低 35 倍，每兆瓦 token 吞吐提升 50 倍。超万名 NVIDIA 员工已在使用 Codex。

【观点】
10 万 GPU 集群的可靠运行对网络提出极高要求——NVL72 机柜内依赖 NVLink，机柜间需 Spectrum-X 高性能以太网或 InfiniBand 互联，低延迟、无丢包的 RoCE/拥塞控制算法是关键。系统软件层面，大规模集群的多节点调度、故障恢复和推理引擎优化（如 Dynamo 配合）决定了 35 倍成本优势能否落地。

> 参考：[OpenAI's New GPT-5.5 Powers Codex on NVIDIA Infrastructure](https://blogs.nvidia.com/blog/openai-codex-gpt-5-5-ai-agents/)

## 2. NVIDIA Dynamo 全栈优化面向 Agentic 推理场景

【快讯内容】
NVIDIA 发布 Dynamo 针对 Agentic 推理的全栈优化方案，在前端 API、路由层和 KV Cache 管理三层实现 agent 原生支持。引入 nvext.agent_hints 结构化信号，让编排器做出更智能的路由和缓存决策，最大化跨 worker 的 KV Cache 复用率。

【观点】
Dynamo 的 KV Cache 路由与复用机制是推理引擎内核优化的前沿方向，直接影响 vLLM/SGLang 等框架在多节点部署中的效率。agent_hints 机制为系统软件提供了新的调度抽象——编排器不再将请求视为匿名 token 流，而是携带上下文的有状态会话，这对多芯片/多节点调度架构有重要参考价值。

> 参考：[Full-Stack Optimizations for Agentic Inference with NVIDIA Dynamo](https://developer.nvidia.com/blog/full-stack-optimizations-for-agentic-inference-with-nvidia-dynamo/)

## 3. 防御 Agentic 环境中的间接 AGENTS.md 注入攻击

【快讯内容】
NVIDIA AI 红队披露 Codex 中通过恶意依赖注入 AGENTS.md 指令的攻击链，涵盖依赖设置、指令优先级滥用和摘要覆盖等环节。文章分析了 agent 指令文件如何扩大传统提示注入的攻击面，并提出缓解策略。

【观点】
该研究直接关联运行时安全与沙箱/进程隔离领域。AI agent 在开发环境中拥有文件读写和命令执行权限，依赖供应链攻击可绕过安全边界。对系统软件而言，需在 agent 运行时层面引入更细粒度的权限控制和指令来源验证机制，NVIDIA OpenShell 等安全沙箱方案的重要性进一步凸显。

> 参考：[Mitigating Indirect AGENTS.md Injection Attacks in Agentic Environments](https://developer.nvidia.com/blog/mitigating-indirect-agents-md-injection-attacks-in-agentic-environments/)

## 4. DeepStream 9 编码代理构建视觉 AI 流水线

【快讯内容】
NVIDIA DeepStream 9 引入编码代理（Coding Agents），支持通过自然语言提示自动生成完整的视觉 AI 流水线，包括 REST API、健康监控、部署自动化和 Kafka 集成，大幅降低多摄像头实时视频分析应用的开发门槛。

【观点】
DeepStream 是嵌入式/边缘 AI 部署的核心框架，Coding Agent 模式降低了 Jetson 等平台上视觉流水线的开发成本。从嵌入式软件角度看，自动生成的流水线代码仍需关注实时性约束和资源优化——生成代码能否满足边缘设备的内存和算力限制、是否适配 RTOS 调度需求，是落地关键。

> 参考：[How to Build Vision AI Pipelines Using NVIDIA DeepStream Coding Agents](https://developer.nvidia.com/blog/how-to-build-vision-ai-pipelines-using-deepstream-coding-agents/)

## 5. Omniverse 模块化库集成 Physical AI 能力至现有应用

【快讯内容】
NVIDIA 将 Omniverse 核心组件（RTX 渲染、PhysX 仿真、数据存储）封装为独立 C API 库（ovrtx、ovphysx、ovstorage），支持 C++ 和 Python 绑定。开发者无需采用完整 Omniverse 容器栈，即可将渲染、物理仿真能力嵌入已有服务。

【观点】
库优先架构对嵌入式软件和系统软件均有意义。PhysX 独立库可直接用于机器人仿真验证流程，降低 Jetson 边缘端数字孪生开发的集成成本。系统软件层面，headless-first 设计和独立进程调用模式有利于在多芯片/分布式环境中灵活部署仿真服务，避免单体运行时的资源开销。

> 参考：[Integrate Physical AI Capabilities into Existing Apps with NVIDIA Omniverse Libraries](https://developer.nvidia.com/blog/integrate-physical-ai-capabilities-into-existing-apps-with-nvidia-omniverse-libraries/)

## 6. 端到端 FP8 精度加速强化学习训练

【快讯内容】
NVIDIA 展示在 GRPO 强化学习训练中使用端到端 FP8 精度的方案，对比 BF16 基线、仅生成阶段 FP8、以及生成+训练全 FP8 三种配方，在保持模型质量的前提下显著提升训练吞吐量。

【观点】
FP8 端到端训练方案直接关联系统软件中的推理引擎内核优化方向。GB200/GB300 的 FP8 Tensor Core 能力需要驱动和运行时层面的精确支持——包括精度管理、缩放策略和内存调度。对于多节点 RL 训练场景，generation 和 training 阶段的混合精度切换还涉及跨节点通信的数据格式协调。

> 参考：[Run High-Throughput Reinforcement Learning Training with End-to-End FP8 Precision](https://developer.nvidia.com/blog/run-high-throughput-reinforcement-learning-training-with-end-to-end-fp8-precision/)
