## 一、CUDA 内核融合：以更少访存与更少启动榨干 GPU 带宽

【快讯内容】
文章系统讲解内核融合：GPU 算力远超显存带宽，把多个算子合并进单一 device kernel，可让中间结果留在寄存器/共享内存而不必往返全局显存，同时省去逐次 kernel 启动开销。给出手写融合、CUDA 编译期融合与库级融合等多条落地路径，直击访存受限与启动受限两类瓶颈。

【观点】
面向系统软件：这是推理引擎内核优化最基础也最高杠杆的手段。逐 token 解码阶段大量小算子（LayerNorm、逐元素激活、偏置加）正是访存/启动双重受限，融合能显著抬高算术强度与 SM 占用，与 vLLM/SGLang/TensorRT 的 CUDA Graph、算子编译方向直接互补；"中间结果不回全局显存"也呼应 KV-cache 与显存带宽压力治理。面向网络/嵌入式：本文聚焦单卡 kernel 层，网络无关；但端侧 Jetson 显存带宽更紧张，融合对边缘推理时延同样有直接参考价值。

> 参考文章：[Kernel Fusion in NVIDIA CUDA: Optimizing Memory Traffic and Launch Overhead](https://developer.nvidia.com/blog/kernel-fusion-in-nvidia-cuda-optimizing-memory-traffic-and-launch-overhead)

## 二、模型—硬件协同设计：面向 GPU 友好的 LLM 结构

【快讯内容】
文章提出以硬件感知方式设计 Transformer：线性层维度趋近正方形、对齐 GPU tile 尺寸、采用"宽而浅"（width-over-depth）的宽高比，以最大化算术强度与 GPU 利用率，从而在高吞吐、低时延推理下取得更优表现。把模型结构选择从纯精度视角转为软硬件协同的系统视角。

【观点】
面向系统软件：这是"模型即部署目标"的协同设计思路，对推理引擎调度极具指导性——维度对齐 tile、宽高比适配 GEMM，直接决定 kernel 能否打满 Tensor Core，是继量化/融合之后的结构级优化抓手，可与大专家并行、多 token 预测叠加。面向嵌入式：宽而浅结构在端侧同样利于降低逐层串行时延、契合有限显存下的算术强度优化，对 Jetson 部署有参考价值。面向网络：多节点切分时近正方形层更易均衡张量并行通信，对拓扑映射有间接意义。

> 参考文章：[AI Model Co-Design: Hardware-Friendly LLM Design](https://developer.nvidia.com/blog/ai-model-co-design-hardware-friendly-llm-design)

## 三、JAX 训练主机卸载：把激活搬到固定主机内存以缓解 HBM 瓶颈

【快讯内容】
针对大模型训练中 HBM 容量/带宽吃紧，文章介绍 JAX 的 host offloading：将前向选定的激活转存到 pinned host memory，反向传播时再流式取回，从而显著降低 HBM 压力、支撑更大批量或更长序列，同时以 PCIe/NVLink C2C 异步拷贝隐藏搬运开销。

【观点】
面向系统软件：这是训练侧的显存—主机分层调度，与推理侧 KV-cache offload 思路同源。pinned memory + 异步流式取回的核心是"用 CPU-GPU 互联带宽换 HBM 容量"，其收益强依赖拷贝与计算的重叠调度，对运行时的内存生命周期管理与流水编排是直接参考。面向网络：卸载把压力转到 CPU-GPU 互联关键路径，Grace-Vera C2C/NVLink 高带宽低时延成为该方案成立前提，与近期 Vera CPU 内存带宽篇形成呼应。面向嵌入式：端侧统一内存架构（Jetson）本身即 host-device 共享，本文的重叠隐藏思路对边缘大模型内存治理有间接借鉴。

> 参考文章：[Reducing High-Bandwidth Memory Bottlenecks in JAX-Based LLM Training with Host Offloading](https://developer.nvidia.com/blog/reducing-high-bandwidth-memory-bottlenecks-in-jax-based-llm-training-with-host-offloading)
